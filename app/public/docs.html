Documentação

O NexusGate é um gateway de LLM compatível com o formato da API da OpenAI. Você troca a base_url da sua aplicação e usa uma chave do NexusGate — nós roteamos para o modelo mais barato que resolve, reaproveitamos respostas com cache e usamos a sua própria chave do provedor (BYOK).

Visão geral

Existem dois planos de acesso, com credenciais diferentes:

  • Plano de dados (suas aplicações): autentica com uma chave do NexusGate no header x-api-key. É por aqui que passam as chamadas de IA.
  • Painel (você/seu time): login via Supabase para gerenciar chaves, credenciais BYOK, ver uso e cobrança.
Base URL: https://api.nexusgate.com.br/v1 (produção) — em desenvolvimento, http://localhost:8000/v1.

Autenticação

Toda chamada ao proxy exige o header x-api-key com uma chave gerada no painel (formato nxg_....). A chave em claro é mostrada uma única vez, na criação.

x-api-key: nxg_a1b2c3d4.seu-segredo-aqui

Sem a chave → 401. Chave inválida/revogada → 403.

Quickstart

Use qualquer SDK compatível com OpenAI — só muda a base_url e a chave:

from openai import OpenAI client = OpenAI( base_url="https://api.nexusgate.com.br/v1", api_key="nxg_a1b2c3d4.•••••", # sua chave do NexusGate ) resp = client.chat.completions.create( model="nexus-auto", # roteamento automático por complexidade messages=[{"role": "user", "content": "Escreva testes para esta função"}], ) print(resp.choices[0].message.content)

Chat Completions plano de dados

POST /v1/chat/completions

Proxy compatível com o formato OpenAI. Resolve a credencial BYOK do tenant, chama o provedor real (qualquer LLM, na nuvem ou local) e devolve a resposta. Suporta streaming (SSE) com "stream": true.

Campos: model, messages, provider (opcional — inferido do modelo), stream, max_tokens, temperature, fallback.

curl https://api.nexusgate.com.br/v1/chat/completions \ -H "x-api-key: nxg_..." -H "Content-Type: application/json" \ -d '{ "model": "gpt-4o-mini", "messages": [{"role": "user", "content": "Olá!"}] }'

Roteamento — nexus-auto

Ao enviar "model": "nexus-auto", o gateway classifica a complexidade da requisição e escolhe a rota mais econômica, com política local-first + escalonamento:

  • Local é o primeiro recurso (gratuito), para tarefas simples e complexas.
  • Se o local não der conta (falha/indisponível), escala para um modelo pago — o premium no caso de tarefa complexa.
  • Sem provedor local, escolhe direto o hospedado mais adequado ao tier.

Cada resposta traz a decisão nos headers (x-nexus-model, x-nexus-complexity, x-nexus-routed) e a economia é registrada em Uso & Economia.

Fallback

Defina uma cadeia de reserva: se o modelo principal falhar, o gateway tenta os próximos automaticamente.

{ "model": "gpt-4o-mini", "messages": [...], "fallback": ["google:gemini-2.5-flash", "ollama:qwen3.5:4b"] }

Cada item é provider:model ou apenas model (o provedor é inferido).

Cache semântico

Requisições semanticamente parecidas não pagam duas vezes. Usamos embeddings para comparar o prompt com respostas anteriores do tenant; se a similaridade passar do limite, servimos a resposta do cache em milissegundos — sem chamar o provedor. O header x-nexus-cache indica hit ou miss, e a economia entra no relatório.

Configurável por tenant: limite de similaridade e TTL. Prompts diferentes não colidem.

Modelos plano de dados

GET /v1/models

Lista, no formato OpenAI, os modelos disponíveis nos provedores que o seu tenant cadastrou (consulta cada provedor em tempo real).

Embeddings plano de dados

POST /v1/embeddings

Gera embeddings via BYOK (OpenAI-compatível ou Ollama). Retorna no formato OpenAI.

curl https://api.nexusgate.com.br/v1/embeddings \ -H "x-api-key: nxg_..." -H "Content-Type: application/json" \ -d '{"provider": "openai", "model": "text-embedding-3-small", "input": "olá"}'

Credenciais BYOK painel

Você cadastra as chaves dos seus provedores (OpenAI, Anthropic, Gemini, Qwen…) ou aponta para um modelo local (Ollama, LM Studio, vLLM). Elas ficam cifradas em repouso com AES-256-GCM (envelope encryption) e só são descriptografadas em memória, no instante da chamada.

GET /v1/admin/provider-keys
POST /v1/admin/provider-keys

Campos: provider, api_key (opcional para locais), base_url (opcional), format (openai/anthropic/ollama), label, default_model.

Chaves & limites (chaves virtuais) painel

POST /v1/admin/keys
PATCH /v1/admin/keys/{id}

Crie chaves de API para suas apps e defina limites por chave: orçamento mensal (USD), rate limit (rpm) e allowlist de modelos. Uma chave só consegue usar os modelos permitidos e para de funcionar ao atingir o orçamento.

Uso & Economia painel

GET /v1/admin/usage/summary
GET /v1/admin/logs

O resumo mostra tokens e custo por LLM, com o comparativo "se tudo rodasse no mesmo LLM" (economia auditável). Os logs trazem cada requisição (modelo, tokens, custo, latência, status, cache). Prévias de prompt/resposta são opt-in e sempre com redação de PII (LGPD).

Planos & Cobrança painel

GET /v1/admin/billing

Planos em BRL com quota mensal e rate limit por plano. Cobrança da orquestração — o consumo dos modelos é pago direto na sua chave (BYOK), sem markup sobre tokens.

Headers de resposta

  • x-nexus-request-id — id da requisição (rastreio nos logs).
  • x-nexus-model / x-nexus-provider — modelo e provedor efetivamente usados.
  • x-nexus-cache — hit ou miss.
  • x-nexus-complexity / x-nexus-routed / x-nexus-local — decisão do nexus-auto.

Segurança & LGPD

  • BYOK cifrado em repouso (AES-256-GCM); chaves nunca em logs.
  • Redação de PII (CPF, CNPJ, e-mail, telefone, cartão) nos logs e, opcionalmente, antes de enviar a provedores hospedados.
  • Isolamento por tenant, rate limit e quotas.
  • Proteção anti-SSRF nos endpoints de provedor.
